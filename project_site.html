01 · Research Question

왜 이 연구인가, 무엇을 묻는가

한국의 연간 최대전력은 매년 7~8월 오후에 발생합니다. 특히 도심의 전기 냉방 부하는 전력망에 큰 부담이며, 이를 완화할 수단으로 ESS(Energy Storage System, 에너지저장장치)가 주목받습니다. 그런데 "어디에 얼마짜리 ESS를 설치해야 효과적인가"는 도시 단위의 정량 분석이 필요합니다.

초기에 설정한 연구질문

  • RQ1. 마곡동의 건물별 전력 수요를 머신러닝으로 예측할 수 있는가?
  • RQ2. 어느 시점·어느 건물이 피크 부하를 형성하는가?
  • RQ3. 공용 ESS를 두면 마곡동 피크는 얼마나 완화되는가?
  • RQ4. 위 결과를 도시 단위 3D 디지털트윈으로 통합 시각화할 수 있는가?
생각의 출발

처음에는 "건물 크기·층수·높이가 에너지를 결정한다"는 직관적 모델을 세우고 시작했습니다. 그런데 데이터를 직접 까보면서 그 가설이 무너졌고, 그 무너짐 자체가 이 프로젝트의 핵심 발견이 되었습니다. 아래는 가설이 어떻게 수정되어 갔는지의 사고 과정을 그대로 따라간 기록입니다.

전체 연구 흐름

데이터 통합→ 탐색·발견→ ML 예측→ 군집화→ ESS 시뮬레이션→ 3D 디지털트윈

왜 좌표 식별자인가

건물명이 11,358동 중 약 절반(6,076동)만 존재합니다. 건물명을 1차 키로 쓰면 절반은 식별 불가. 좌표는 100% 채워져 있어, 위경도를 반올림한 키 "x_y"를 식별자로 채택했습니다. 같은 좌표가 동일 건물의 다른 세대/부분일 수 있어, 같은 키의 행들은 에너지는 합산, 특성은 첫 값으로 집계했습니다.

03 · Key Findings

데이터가 알려준 네 가지 발견

분석은 계획대로 흐르지 않았습니다. 데이터를 직접 까볼수록 가설이 흔들렸고, 그 흔들림이 새로운 방향을 만들었습니다.

Finding 01

'음영도' 컬럼은 사실 자치구 코드였다

⚡ Moment: 음영도 고유값을 세어보니 정확히 25개

처음에는 태양광 분석용 음영도(0~1 또는 0~100% 비율)로 알고 받았습니다. 그런데 분포를 보니 11110, 11140, 11170... 11740 사이의 25개 고유값. 서울시 25개 자치구의 법정동코드였습니다.

  • 11110 = 종로구, 11140 = 중구, ..., 11500 = 강서구(마곡동 소속)
  • 받은 자료가 마곡동만이 아니라 서울시 전체 건물 데이터였음.

→ 영향: 강서구(11500)로 1차 필터 → '마곡' 키워드 건물의 좌표 분포(1~99분위)로 경계 근사 → 약 11,358동의 마곡 데이터 확보. 그런데 이는 행정경계가 아닌 통계적 근사로, 한계로 명시.

Finding 02

건물 특성으로는 에너지를 예측할 수 없다

⚡ Moment: 면적 ~ 에너지 상관 r = 0.07

"큰 건물이 에너지를 많이 쓴다"는 가장 직관적인 가설은 데이터에서 무너졌습니다.

변수에너지와 상관해석
건물면적r = 0.07사실상 무관
층수r = 0.03사실상 무관
건물높이r = 0.03사실상 무관
그때 생각

"이 에너지 값이 진짜 측정값이 맞나? 면적·층수와 무관할 리가 있나?" — 데이터의 정체를 의심. 서울시 API 명세를 확인하면 풀릴 문제지만, 정확한 명세를 못 찾아 "이 데이터의 산출방식은 불명확하다"는 한계를 인정하고, 다음 단계로 넘어갔습니다. 다만 이게 ML 입력 변수를 완전히 바꾸는 결정으로 이어졌습니다.

→ 영향: ML 입력을 "건물 특성"에서 "기상"으로 전환. 목표도 "건물 개별성 예측"이 아닌 "도시 단위 계절 예측"으로 재정의.

Finding 03

에너지는 사실상 '기온(계절)' 하나로 설명된다

⚡ Moment: 기온 ~ 에너지 상관 r = 0.70 (압도적)

기상 변수로 입력을 바꾸자 신호가 분명해졌습니다. 마곡의 월별 평균 에너지는 1월 17.8 → 6월 143.3(피크) → 12월 20.9의 여름 단봉형 패턴.

기상 변수에너지와 상관
기온r = 0.70
강수r = 0.59
습도r = 0.40
일조r ≈ 0.06
풍속r ≈ 0.01

→ 영향: 한국 건물의 통상 U자형(겨울 난방 + 여름 냉방)과 달리 여름만 높은 단봉형 → 이 데이터는 전기 위주(난방=가스 누락)로 추정. ESS도 냉방 피크 대응에 초점.

Finding 04

강수가 1위 변수로 나왔지만, 그건 가짜 신호였다

⚡ Moment: 기상 변수 간 상관 0.79~0.89 (다중공선성)

XGBoost 첫 결과에서 강수가 변수 중요도 1위로 나와 당황. "비가 와서 전기를 더 쓴다?"는 비현실적. 변수 간 상관표를 보니 기온·습도·강수가 모두 서로 0.8 이상의 상관 — 셋 다 "여름"이라는 같은 신호를 중복으로 담고 있었습니다.

기온습도강수에너지
기온1.000.790.850.70
습도0.791.000.890.40
강수0.850.891.000.59

→ 영향: 강수·습도를 빼고 기온 단독으로 모델 재검증 → 성능 거의 동일(R² 0.668 vs 0.668). 즉 강수의 "중요도"는 기온과 정보를 나눠 가진 그림자였음. 최종 변수는 CDD + HDD + 면적으로 단순화하며 해석 가능성 확보.

가장 정직한 모델이 가장 잘 일반화됐다."

04 · Methodology

분석 방법론의 설계

단순히 "어떤 알고리즘을 썼다"가 아니라, "왜 그 평가 방식을 골랐는가"를 함께 설명합니다. 학부 프로젝트지만 평가 설계의 정직성이 이 연구의 신뢰도를 결정한다고 보았기 때문입니다.

전체 파이프라인

  1. 병합: 마곡 건물(좌표·면적) × 월별 에너지(1~12월) × 월별 기상(5종) → 건물×월 long format (136,296행).
  2. 탐색: 변수별 상관, 분포, 결측·중복 점검. 여기서 위의 발견 4개가 도출됨.
  3. ML 예측: 건물단위·월단위·무작위 3가지 분할로 R² 비교 → 가장 정직한 평가 선택.
  4. 군집화: 12개월 패턴 정규화 후 K-Means → 강도 기반 3개 군집.
  5. ESS 시뮬레이션: 시간별 표준 프로파일 가정 + peak shaving 알고리즘 → 용량별 저감률.
  6. 3D 디지털트윈: Cesium OSM Buildings 위에 격자 매칭으로 색상·이펙트 부여 → 인터랙티브 시뮬레이션.

평가 설계 — 같은 모델, 다른 분할

05 · Machine Learning

머신러닝 모델링과 검증

모델 선택

트리 기반 부스팅 계열을 선택. 선형 회귀 대비 비선형 관계 포착 가능하고, SHAP 등 해석 도구가 풍부하며, 학부 프로젝트 범위에서도 안정적이기 때문.

  • 주 모델: XGBoost (n_estimators=300, max_depth=4, learning_rate=0.08)
  • 비교군: sklearn GradientBoostingRegressor (인터넷 차단 환경에서 검증용), 선형회귀(baseline)
  • 특성: CDD, HDD, 건물면적, 월(원-핫 미사용, 정수 그대로)
  • 대상: 월별 에너지사용량 (회귀)

  • 건물단위 5-fold R² = 0.691 (±0.004) — 처음 보는 건물도 기상 정보로 약 69% 설명.

정직한 해석

R² 0.69라는 숫자는 "꽤 괜찮아 보이지만", 그 설명력의 대부분은 건물 특성이 아니라 기상(계절)에서 옵니다. 즉 이 모델은 "처음 보는 건물을 예측"하는 것이 아니라, 사실상 "1월이면 낮고 6월이면 높다"는 계절 패턴을 적용하는 것에 가깝습니다. 이게 약점이라기보다, 이 데이터로 가능한 가장 정직한 결론입니다. "건물 개별성 예측은 어렵다"는 발견이 다음 단계(군집화)로 자연스럽게 연결됩니다.

06 · Clustering

건물 군집화 — 강도의 차이만 남다

건물 특성으로 개별 예측이 안 된다면, 12개월 패턴 모양으로 묶을 수 있을까? 각 건물의 12개월 에너지를 합=1로 정규화(모양만 비교) 후 K-Means 적용.

왜 정규화부터 했나

총량으로 묶으면 그냥 "큰 건물 vs 작은 건물"로 갈려서 계절 모양이 안 보입니다. 우리가 보고 싶은 건 "여름피크형 vs 겨울피크형 vs 평탄형" 같은 패턴 모양이므로, 각 건물의 12개월을 비율(합=1)로 정규화한 뒤 묶어야 했습니다.

k 결정 — 실루엣 vs 해석

kInertiaSilhouette해석
210.50.616"여름형/덜여름형" — 너무 단순

실루엣은 k=2가 가장 높지만, 이는 데이터가 단봉형 단일 패턴이라 둘로 가르기 쉽기 때문. 해석 가능성을 위해 k=3 채택: 여름 집중도가 단계적으로 나뉘고, ESS 우선순위 적용에 자연스러움.

예상과 달랐던 점

"여름피크형 vs 겨울피크형 vs 평탄형" 같은 질적 유형 차이를 기대했지만, 마곡 건물은 모두 동일한 여름 단봉 패턴(피크월=6월). 차이는 강도(여름에 얼마나 몰리느냐)뿐.

겨울 비중도 모든 군집에서 0.04~0.11로 매우 낮음 → 이 데이터엔 난방 피크형 건물이 거의 없음. 이는 발견 ③(데이터가 전기 위주, 난방=가스 누락)을 다시 확인.

최종 군집 (강도 기반 k=3)

군집건물수여름비중평균 연총량특성·전략
냉방집중형1,1310.54507여름에 가장 몰림 → ESS 우선 대상
중간형3,0360.47838중간
분산형7,1910.411,079연중 비교적 고름

흥미로운 사실: 여름 집중도와 연총량이 반비례. 큰 건물일수록 연중 꾸준히 쓰고, 작은 건물일수록 여름에 반짝 몰리는 구조. "작은 건물이 사실은 ESS 적용 효과가 가장 크다"는 정책적 시사점이 가능.

07 · ESS Scenario

ESS 시나리오 설계와 시뮬레이션

설계 결정 — 두 갈래에서 하나를 선택

데이터가 월별 에너지(kWh)만 가지고 있어, ESS 시뮬레이션 설계에서 두 갈래를 두고 고민했습니다.

경로방식장점단점
길 1월 단위 부하 평준화가진 데이터로 정직하게 계산"peak shaving" 통상 의미와 거리
길 2 (채택)시간별 표준 프로파일 분해진짜 일중 peak shaving 표현 가능프로파일이 가정 → 한계 명시 필요
왜 길 2를 택했나

발표 임팩트와 디지털트윈 시각화 가능성 때문. 시간별 곡선이 있어야 "낮 피크를 깎는다"는 직관적 데모가 됩니다. 대신 "프로파일은 외부 가정"이라는 한계를 발표·논문 모두에서 명확히 표시.

정책적 시사점

마곡 같은 신도시 단위에서 일에너지 10% 규모 공용 ESS를 두면 여름 피크의 약 1/5을 깎을 수 있다는 참고 수치를 얻을 수 있습니다. 단, 시간별 프로파일이 가정이라는 한계 하에서의 추정치임을 반드시 동반 표기.

08 · Digital Twin

3D 디지털트윈 구현

결과를 정적인 그래프로 끝내지 않고, 실제 마곡동 지도 위에서 만져볼 수 있는 인터랙티브 시뮬레이션으로 통합. 분석 → 통찰 → 데모가 하나의 화면에서 끝나는 것이 이 프로젝트의 정체성.

기술 스택 선택

후보장점단점채택

좌표계 문제 — 추측을 사실로 교정

초기 EPSG:5186(GRS80 중부원점)으로 추정 변환했으나 건물이 엉뚱한 곳에 표시됨. 역변환 수식으로 한국 주요 좌표계 후보들을 모두 시도한 결과 EPSG:5179 (UTM-K)가 정답. UTM-K로 변환하면 마곡동(경도 126.82, 위도 37.56)에 정확히 위치.

# UTM-K (EPSG:5179) 역변환 파라미터
lon0 = 127.5;  lat0 = 38
k0   = 0.9996; FE = 1000000;  FN = 2000000
ellipsoid = "GRS80"

색상 매칭 전략 — 격자 기반

OSM 건물 하나하나에 우리 건물 데이터를 1:1로 연결하는 것은 불가능(키가 다름). 대신 마곡동을 약 50m × 50m 격자로 나누고, 각 격자의 대표값(군집·평균에너지·여름비중)을 계산. Cesium이 화면의 OSM 건물 위치를 격자에 매칭해 색을 입힘.

  • 장점: 전체 OSM 건물에 빈틈없이 색 부여, 성능 양호(함수 기반 색 계산)
  • 한계: 건물 단위 1:1 매칭이 아닌 격자 단위 근사

디지털트윈 핵심 기능

📅

월 슬라이더 + 1년 재생

1~12월 슬라이더로 계절 변화 시각화. 1년 재생 버튼으로 자동 순환. 각 건물의 그 달 실측 에너지가 색으로 표현됨.

🌡️

기온 시나리오 (-5 ~ +10°C)

ML 회귀 계수(CDD/HDD)를 JS에 심어 실시간 재계산. "온난화 +3°C 시" 마곡 피크 변화를 즉시 확인.

🔋

ESS 설치 토글 + 시설 모델

버튼 클릭 시 마곡 중앙에 배터리 컨테이너 6개 + 관제타워 + 후광이 솟아오르는 ESS 단지 생성.

⚡

충방전 이펙트 + 빛줄기

충전 시 파란빛 수축, 방전 시 노란빛 확장 + 주변 피크 건물 14개로 흐르는 빛줄기. PolylineGlow로 "전기가 흐르는" 시각 효과.

📊

실제 시뮬레이션 곡선 연동

용량 슬라이더(1~30%)가 실제 ESS 시뮬레이션 저감률 곡선 (4.7% → 38.5%)을 그대로 따름. 발표 숫자와 디지털트윈 숫자가 일치.

09 · Limitations & Future Work

정직하게 밝히는 한계

학부 프로젝트 범위에서 도달한 정직한 한계들. 가리지 않고 밝히는 것이 분석의 신뢰도를 높인다고 봅니다.

① 시간 해상도

원자료가 월별 에너지(kWh)뿐 → 시간별 부하는 외부 표준 프로파일을 가정으로 합성. 따라서 peak shaving 저감률은 "실측"이 아니라 "이 가정 하에서의 추정"입니다. 실제 시간별 데이터(한전·전력거래소 등) 확보 시 곧바로 정밀화 가능.

② 에너지 단위 불명

서울시 API 원자료의 정확한 단위·산출방식이 확정되지 않아, 절대값(kW) 대신 상대적 저감률(%)로 제시. 값의 내적 일관성(월별합 = 총량)은 검증됨.

③ 마곡동 경계 근사

건물명에 '마곡'이 포함된 건물 좌표 분포 1~99분위로 경계 근사. 행정구역 경계가 아님. 향후 정밀한 행정경계 shapefile 사용 시 보완 가능.

④ 용도 분류의 한계

건물명 기반 자동분류 결과 전체의 약 40%만 식별 가능(절반은 건물명 자체가 없음). 이에 ESS는 전 건물 적용되는 군집 기준을 메인으로, 용도는 보조로만 사용.

⑤ 시계열 일반화의 한계

월별 데이터가 12개월뿐이라, 월단위 분할 평가에서 R²가 거의 0. 즉 학습 안 한 계절을 예측하는 능력은 입증 못함. 다년치 데이터 확보 시 시계열 모델로 확장 가능.

⑥ OSM 건물 매칭의 근사성

디지털트윈의 실제 건물(Cesium OSM)에 우리 데이터를 입힐 때 위치 기반 격자 매칭을 사용. 건물 단위 1:1 정확 매칭은 아니며, 향후 서울시 3D 건물 데이터(CityGML) 연계 시 개선 가능.

결론 — 한 문장으로

마곡동은 냉방 의존형 에너지 구조라 여름 피크가 뚜렷하며, 중앙 공용 ESS(일에너지 10% 규모)로 피크를 약 21% 완화할 수 있다. 이 모든 분석은 디지털트윈 위에서 인터랙티브하게 검증된다.
